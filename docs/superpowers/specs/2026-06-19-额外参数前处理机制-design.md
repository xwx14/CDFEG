# 设计文档：有限元额外参数的前处理读取机制（以 DEl2D Newmark gamma/beta 为首个应用）

> 日期：2026-06-19
> 范围：CDFEG 核心库（C++）+ pyTool 代码生成器（Python/Jinja2）+ sample/DEl2D 示例接入
> 状态：待用户审查 → 转 writing-plans

---

## 一、背景与目标

CDFEG 当前把"算法参数"分两类处理：
- **网格/材料/边界**：由 GiD 前处理 → `.dat` → `GidPrePost::pre()` 读取，已完备。
- **算法级额外参数**（如 Newmark-β 的积分参数 `gamma/beta`）：**硬编码在示例源码**中（`NewmarkData.cpp:50` 的 `setNewmarkParams(0.5, 0.25, _dt)`）。

**目标**：建立一条**通用的"额外参数前处理读取"链路**，让任意层（总体数据类 / 物理场 / 单元）能声明"我需要从前处理读取哪些参数"，并在运行时由 `GidPrePost` 自动从 `.dat` 读取、回填给声明者。DEl2D 的 Newmark `gamma/beta` 作为首个落地用例，替换硬编码。

> 备注：`dt/tMax` **已经**从 `.dat` 的 `time` 段读取（`gidPrePost::readTime` → `_femData->_dt/_tMax`），不在本次改造范围；本次仅前处理化 `gamma/beta` 等算法积分参数。

## 二、现状分析

### 2.1 C++ 侧：`_addParams` 是预留未接通字段

`ElementBase.h`、`PhyFieldData.h`、`FemData.h` 三层均已声明：

```cpp
// 需要从前处理输入的参数，每组参数第一个为此参数的组名，后面为参数名
std::vector<std::vector<std::string>> _addParams;
```

但在 `.cpp` 中**零读写**——是纯预留接口。本次正是把它接通。

### 2.2 `time` 段的既有机制（本次的范式参照）

- `.bas` 模板输出：`** name=time,structure="F64"` + 一行 `*GenData(TimeStep) *GenData(TotalTime)`。
- `.prb` 模板定义项目级 GenData：`QUESTION: TimeStep / VALUE: 0.1`、`QUESTION: TotalTime / VALUE: 1.0`。
- 运行时 `gidPrePost::pre()` 按 `name=="time"` 命中 → `readNextLine()` → `TextReader::splitDoubles` → 写入 `_femData->_dt/_tMax`。

这是一条**项目级 GenData**链路：定义在 `.prb`、经 `.bas` 用 `*GenData()` 输出到 `.dat`、由读取器按一行 double 解析。

### 2.3 DEl2D 现状

- `main.cpp` 为 GiD 文件形式（`setFilePath` + `pre()` 读 `.dat`），但**无 `.gid` 目录、无 `.dat` 文件**——配套 GiD 问题类型文件尚未生成。
- Newmark 参数：`dt/tMax` 走 `time` 段；`gamma=0.5/beta=0.25` 在 `NewmarkData.cpp:50` **硬编码**。
- 升级说明 5.11：**不可运行 `testDEL2D.py`**（会以 `ElQ4g` 名覆盖手写 `NewmarkQ4g` 的 C++ 源码）。本次严守此约束。

### 2.4 pyTool 侧

- `DataProject`/`DataField`/`DataEleSub` 已新增 `self.preParams=[]`（结构 `{"name":"组名","params":[参数名]}`），`toDict`/`fromDict` 已补全。
- `MakerGidFile._initFromProject` 构造 `self.data` 字典传给 Jinja2 模板；`gidbas.j2`/`gidprb.j2`/`gidcnd.j2` 为目标模板。

## 三、设计决策

| # | 决策 | 选择 | 理由 |
| --- | --- | --- | --- |
| 1 | `.dat` 中额外参数格式层级 | **项目级 GenData，完全参照 `time`** | 与既有 `time` 同构，链路最短、最 KISS |
| 2 | 读取后值的存储/访问模型 | **各层自持容器 + `gidPrePost` 回填** | 与 `_addParams` 三层声明对称，符合 SRP；各层就近取值，不依赖集中查询 |
| 3 | 本次产出边界 | 机制 + DEl2D 接入 + 手写 `.dat` 验证；**不跑 `testDEL2D.py`** | 完整可用且不违反升级说明 5.11 |
| 4 | GiD 模板机制 | 改 `gidbas.j2` + `gidprb.j2`；**不改 `gidcnd.j2`** | 项目级 GenData 走 `.prb/.bas`；`.cnd`（单元/节点级 Condition）用不上 |
| 5 | `preParams` 结构 | 扩展加 `defaults` 字段 | `.prb` 的 `VALUE` 需默认值，与既有 `paramValues` 风格一致 |

## 四、端到端数据流

```
【1. pyTool 定义层】
DataField.preParams = [{"name":"newmark","params":["gamma","beta"],"defaults":[0.5,0.25]}]
        │  MakerGidFile._initFromProject 汇总 project/field/ele 三层 preParams（按组名去重）→ self.data["preParams"]
        ▼
【2. GiD 模板层】
gidprb.j2：QUESTION: gamma / VALUE: 0.5   QUESTION: beta / VALUE: 0.25   （PROBLEM DATA 内，time 段之后）
gidbas.j2：time 段之后追加
           ** name=newmark,structure="F64"
           *GenData(gamma) *GenData(beta)
        │  GiD 载入问题类型 → 用户填值 → 导出 .dat
        ▼
【3. .dat 文件】（手写样例，数值行直接写值，等价于 GiD 渲染 *GenData 后的结果）
** name=time,structure="F64"
0.1 1.0
** name=newmark,structure="F64"
0.5 0.25
...（coord / mat_NewmarkQ4g / elem / 边界段不变）
        ▼
【4. C++ 运行时 · gidPrePost::pre()】
  ① collectPreParamDecls()：遍历 _femData / 各 phyField / 各 ele 的 _addParams
        → _preParamDecls[组名] = {指向该层 _paramValues 的指针, 参数名列表}
  ② 解析 *name= 段，命中声明表中的组名 → readNextLine → splitDoubles
        → 截断到参数名数 → 回填到「该层」_paramValues[组名]
        ▼
【5. C++ 使用层】
NewmarkDispFieldData 构造：_addParams = {{"newmark","gamma","beta"}}
NewmarkData::caculate：gamma/beta = aField->getParam("newmark", ...) → setNewmarkParams(gamma,beta,_dt)
```

**核心纽带**：组名 + 参数名在四处保持一致——pyTool `preParams[].name`/`params`、`.bas` 的 `*name=` 段名与 `*GenData()` 名、`.prb` 的 `QUESTION` 名、C++ `_addParams` 的组名与参数名。

## 五、详细设计

### 5.1 C++ 核心库（4 文件）

**① 三层各加值容器 + 取值 API**（`_addParams` 不动）：

```cpp
// FemData.h / PhyFieldData.h / ElementBase.h 各加成员：
std::map<std::string, std::vector<double>> _paramValues;   // 组名 → 值（按该组参数名顺序对齐）
double getParam(const std::string& group, const std::string& param) const;  // 命中返回值，否则 0.0
```

`getParam` 实现：在 `_addParams` 中找到首元素等于 `group` 的组，定位 `param` 在其中的下标 `k`（≥1），返回 `_paramValues.at(group)[k-1]`（越界/缺失返回 0.0）。

**② gidPrePost 收集声明表 + 按组名回填**：

```cpp
// gidPrePost.h 新增成员：
std::map<std::string,
         std::pair<std::map<std::string,std::vector<double>>*, std::vector<std::string>>
        > _preParamDecls;   // 组名 → {目标层 _paramValues 指针, 该组完整 _addParams 行(含组名)}

int readPreParams(const std::string& group, const std::string& line);
void collectPreParamDecls();
```

- `collectPreParamDecls()`：遍历 `_femData` → 各 `phyField` → 各 `ele`，对每个 `_addParams` 行登记；**保留名（time/basedata/coord/id/ubf/elem）跳过并 stderr 告警**；同名组后者覆盖并告警（约定组名全局唯一）。
- `pre()` 开头调用 `collectPreParamDecls()`；分发处新增兜底分支（置于所有已知 name/type 分支之后）：

```cpp
else if (_preParamDecls.count(nameLower)) {
    _datReader.readNextLine();
    readPreParams(nameLower, line);
}
```

- `readPreParams`：`splitDoubles` → 截断到「参数名数（组行 size−1）」→ 写入 `(*_preParamDecls[group].first)[group]`。风格与 `readTime`/`readMate` 一致（DRY）。

**③ 边界规则**：值不足按实际数截断；类型错误由 `splitDoubles` 容错；组名冲突/保留名告警不中断。

### 5.2 pyTool（4 文件）

**① `preParams` 结构扩展**（`DataProject.py`/`DataField.py`/`DataEleSub.py`）：增加可选 `defaults` 列表，与 `params` 等长，缺失时补 0.0。`toDict`/`fromDict` 同步带上 `defaults`。

**② `MakerGidFile._initFromProject`**：汇总三层 `preParams`（按组名去重）写入 `self.data["preParams"]`，每项形如 `{"name","params","defaults"}`。

**③ `gidprb.j2`**：在 `time` 的 QUESTION 之后、`materials` 之前追加：

```
{% for pp in preParams %}
{% for p in pp.params %}
QUESTION: {{ p }}
VALUE: {{ pp.defaults[loop.index0] }}
{% endfor %}
{% endfor %}
```

**④ `gidbas.j2`**：在 `time` 段之后、`materials` 段之前追加（不依赖 `bDynamic`，只要 `preParams` 非空即输出）：

```
{% for pp in preParams %}** name={{ pp.name }},structure="F64"
{% for p in pp.params %}*GenData({{ p }}) {% endfor %}

{% endfor %}
```

> `gidcnd.j2` 本次**不改**（额外参数为项目级，非单元/节点级 Condition）。

### 5.3 DEl2D 落地（3 文件 + 1 样例）

- `NewmarkDispFieldData`（构造函数）：`_addParams = {{"newmark","gamma","beta"}};`
- `NewmarkData.cpp:50`：

```cpp
double gamma = aField->getParam("newmark","gamma");
double beta  = aField->getParam("newmark","beta");
if (beta <= 0.0) { std::cerr << "[NewmarkData] beta<=0，无法计算 Newmark 积分常数" << std::endl; return -1; }
aField->setNewmarkParams(gamma, beta, _dt);   // 替换硬编码 0.5/0.25
```

  （`main.cpp` 中 `pre()`（行 33）先于 `caculate()`（行 65）执行，`_paramValues` 在此已就绪。）

- **手写 DEl2D `.dat` 样例**：`time` 段后插入 `** name=newmark,structure="F64"` + 一行 `0.5 0.25`，用于验证读取（修改这两个值应使求解结果变化）。

## 七、风险与边界

- **升级说明 5.11**：全程不运行 `testDEL2D.py`，避免覆盖手写 `NewmarkQ4g` C++ 源码。
- **组名保留冲突**：`collectPreParamDecls` 跳过与既有段名冲突的组名并告警。
- **`beta=0`**：导致 Newmark 积分常数 `a0` 无界；`caculate` 中显式校验 `beta>0`。
- **值不足/类型错**：截断 + 容错，与 `readMate` 一致；不中断求解。
- **不在本次范围**：`testDEL2D.py` 重生成；为 DEl2D 生成 `.gid` 问题类型文件（可后续单独跑 `MakerGidFile` 仅生成 GiD 文件，不触 C++ 源码）。
