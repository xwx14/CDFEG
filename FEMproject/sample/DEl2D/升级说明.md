## 一、升级本质

把旧 newmark 项目（FEPG 生成的 C 动力求解器，SuperLU + 双场）升级为基于 CDFEG 库的现代 C++ 框架示例。核心区别于静力版 El2D 的是：**动力学需要时间步进积分，物理场子类必须重写 `eProgram` 与 `uPhy`**。

| 维度 | 旧 newmark（C） | 新 DEl2D（C++） |
|---|---|---|
| 算法载体 | `newmarka.nfe` 脚本 + `enewmarka.c`/`unewmarka.c` | `NewmarkDispFieldData::eProgram/uPhy` |
| 时间循环 | `newmark.c` 的 `goto l1` 主循环 | `NewmarkData::caculate` 的 `for` 循环 |
| 历史状态 | 全局 `unoda`（u/v/w 三列） | 物理场成员 `_u1/_v1/_w1`（上一步）+ `_u/_v/_w`（当前） |
| 质量/阻尼 | 单元内联计算 `mass=rou*vol`、`damp=rou*alpha*vol` | `NewmarkQ4g::run` 输出 `emass`/`edamp`，物理场组合 |
| 求解 | SuperLU（源码内联） | Eigen `SimplicialLDLT` |

## 二、Newmark-β 算法回顾

求解 `M·ü + C·u̇ + K·u = F`，逐步积分（γ=0.5、β=0.25，平均加速度法，无条件稳定、二阶精度）。

**积分常数**（由 dt、γ、β 决定）：
```
a0 = 1/(dt²·β)        a1 = γ/(β·dt)      a2 = 1/(β·dt)
a3 = 1/(2β) - 1       a4 = γ/β - 1       a5 = dt/2·(γ/β - 2)
a6 = dt·(1-γ)         a7 = dt·γ
```

**有效刚度矩阵**（每步重新装配）：
```
K_eff = K + a0·M + a1·C
```

**有效载荷**（M、C 为集中形式，仅对角）：
```
F_eff = F + M·u1·a0 + M·v1·a2 + M·w1·a3 + C·u1·a1 + C·v1·a4 + C·w1·a5
```

**求解后更新**：
```
w^{n+1} = (u^{n+1} - u1)·a0 - v1·a2 - w1·a3
v^{n+1} = v1 + w^{n+1}·a7 + w1·a6
```

> u1/v1/w1 为上一步的位移/速度/加速度。

```
├── NewmarkData.h / .cpp           # 数据类：caculate() 时间步主循环
├── main.cpp                       # GiD预处理 + 注册 disp/velocity/acceleration/stress 结果项
└── CMakeLists.txt                 # 构建目标 del2d
```

### 3.1 NewmarkQ4g 单元（继承 IsoEleBase）

相比静力版 `ElQ4g`，`run()` 额外输出三个量：
- `estif`：刚度 K = ∫BᵀDB dΩ（与 ElQ4g 完全一致）
- `emass`：集中质量 `∫ rou·vol·Nᵢ dΩ`
- `edamp`：集中阻尼 `∫ rou·alpha·vol·Nᵢ dΩ`（Rayleigh 形式）
- `eload`：体力 `∫ Nᵢ·f dΩ`

> emass/edamp 为**一维向量**（仅对角线），对应旧 a1eq4g2 的 `elump[i]·N` 形式。

### 3.2 NewmarkDispFieldData 物理场（继承 PhyFieldData）

**新增成员**（历史状态，按节点平铺存储，长度 = 节点数×dof）：
```cpp
std::vector<double> _u, _v, _w;     // 当前步位移/速度/加速度
std::vector<double> _u1, _v1, _w1;  // 上一步位移/速度/加速度
double _a0.._a7;                    // Newmark 积分常数
```

**重写 `eProgram()`**（核心，**不能复用基类 `eProgram_el`**）：
1. 清零 `_equSys._data` 与 `_equSys._f`（每步重新装配）
2. 遍历单元，组合有效单元矩阵 `K_eff[i][j] = K[i][j]`，对角加 `emass·a0 + edamp·a1`
3. 组装有效载荷 `_f[inv] += eload + M·(u1·a0+v1·a2+w1·a3) + C·(u1·a1+v1·a4+w1·a5)`
4. **置 `_equSys._bSavedData0 = false`** 后调用边界条件（见第五节）

**重写 `uPhy()`**：
1. 从 `_equSys._rhs` 读当前步位移到 `_u`
2. Newmark 更新 `_w`（加速度）、`_v`（速度）
3. 输出节点结果：`u/v`、`velU/velV`、`accU/accV`、应力
4. **`_u1=_u; _v1=_v; _w1=_w;`** 保存为下一步初值

### 3.3 NewmarkData 数据类（继承 FEMData）

**`caculate()` 内实现时间步主循环**：
```cpp
aField->initMatrix();              // 方程编号 + 稀疏骨架（仅一次）
aField->setNewmarkParams(0.5,0.25,_dt);
for (int it = 0; it < nStep; ++it) {
    aField->eProgram();            // 装配有效矩阵+载荷
    aField->solve();               // Eigen LDLT 求解
    aField->uPhy();                // 更新历史 + 应力恢复
    _prePost->post2(it);           // 输出本步 GiD 结果
}
```

## 四、设计原则体现

| 原则 | 体现 |
|---|---|
| **KISS** | Newmark 公式集中在物理场两个方法，时间循环集中在 caculate，结构清晰 |
| **DRY** | 刚度计算复用 ElQ4g 逻辑；稀疏骨架一次建立全程复用；uEle 应力恢复复用 ElQ4g |
| **SOLID-O** | 动力特性通过**子类重写** eProgram/uPhy 注入，不改库代码（eProgram_el 是稳态算法，本就不该复用） |
| **SRP** | 单元只算单刚/单质/单阻/载荷；物理场管有效矩阵组合与状态更新；数据类管时间循环 |

## 五、易出错的地方（重点）

### 5.1 【最关键】`eProgram_el` 是稳态算法，绝不能复用

`PhyFieldData::eProgram()` 默认转发到 `eProgram_el()`，后者：
- 只装配**刚度**，不处理 `eload`（静力算例右端项恰好为0）
- 没有质量/阻尼组合
- 没有历史状态

Newmark 必须**完全重写 `eProgram`**，组合有效矩阵 `K+a0M+a1C` 并把有效载荷写入 `_f`。这是本升级的核心，也是最容易「图省事调用基类」导致结果全错的地方。

### 5.2 【关键】边界条件缓存的基线陷阱（`_bSavedData0`）

`EquationSystem::applyFirstBCs` 内部有缓存机制：
```cpp
if (!_bSavedData0) { _data0 = _data; _f0 = _f; _bSavedData0 = true; }
_data = _data0; _f = _f0;   // 从基线恢复，再施加边界
```

**问题**：首次调用后 `_bSavedData0=true`，后续每步 `_data/_f` 都会恢复成**首步**的值。但动力学每步 `_f`（有效载荷）都在变，会导致后续步载荷全错。

**解决**：每步 `eProgram` 装配完新矩阵/载荷后，**强制 `_equSys._bSavedData0 = false`**，再调 `applyFirstBCs`，使其用当前步数据重建基线：
```cpp
_equSys._bSavedData0 = false;
_equSys.applyFirstBCs(_nodeBC1s, _ida);
```

> 静力算例只装配一次无此问题；动力学每步装配必须注意。

### 5.3 每步必须清零 `_data` 与 `_f`

稀疏骨架（非零元位置）全程不变，但 `_data`（数值）必须每步清零后重新累加，否则会把所有步的矩阵/载荷叠加。`eProgram` 开头：
```cpp
std::fill(_equSys._data.begin(), _equSys._data.end(), 0.0);
std::fill(_equSys._f.begin(), _equSys._f.end(), 0.0);
```

### 5.4 emass/edamp 是「集中」形式（仅对角）

旧 a1eq4g2 中质量/阻尼是集中质量（lumped），`emass[i]` 只对应自身自由度，**非完整矩阵**。有效矩阵组合时只能加在对角线：
```cpp
effMat[i*k + i] += emass[i]*_a0 + edamp[i]*_a1;   // 正确：仅对角
// effMat[i*k + j] += ...（错误：M/C 无非对角项）
```

有效载荷中 M/C 也按一维向量乘标量，勿当矩阵用。

### 5.5 局部方程号与历史向量的索引对齐

`emass[i]` 的下标 `i` 是单元内局部自由度序号，需正确映射到全局历史向量 `_u1` 的下标：
```cpp
int node = nodeIds[i / _dof];          // i/_dof 是单元内节点序号
int iDof = i % _dof;                    // i%_dof 是自由度序号
int idx = node * _dof + iDof;           // 全局历史向量下标
```

错位会导致用错节点的历史值，结果发散或无意义。

### 5.6 历史向量初始化时机

`_u1/_v1/_w1` 在 `eProgram` 第一次读取前必须已分配。`ensureHistorySize()` 在 `setInitialDisp`/`eProgram`/`uPhy` 入口都调用，保证大小 = `_kVar`。**注意 `setNPts`（在 addNodeEnd 触发）只设 `_kVar`，不分配历史向量**，故需惰性 ensure。

### 5.7 初值（u1/v1/w1）必须设置

Newmark 首步即依赖上一步历史（u1/v1/w1）。默认全零对应「零初值」；若算例有初速度/初加速度，必须在 `caculate` 前调用 `setInitialDisp/Vel/Acc` 设置，否则首步有效载荷缺项。

### 5.8 dt/tmax 来源与 gamma/beta 默认

- `dt/tmax` 来自 dat 的 `time` 段（`FEMData::_dt/_tMax`），`caculate` 校验 `_dt<=0` 报错。
- `gamma/beta` 默认 0.5/0.25（平均加速度法），若需其他格式（中心差分法 β=0）需调 `setNewmarkParams`。**β=0 时 a0 无界，dt 受稳定性限制**，需谨慎。

### 5.9 dat 材料名必须匹配单元名

`GidPrePost::readMate` 按 `mat_<单元_name>` 匹配材料：
- 单元 `_name = "NewmarkQ4g"` → dat 材料段必须写 `* name=mat_NewmarkQ4g, type=mat`
- dat 的 elem 段 `name` 也必须是 `NewmarkQ4g`

名称不符会导致材料读不到（返回 -1），单刚/质量全为 0，求解异常。

### 5.10 输出结果项的列名必须与 uPhy 写入的一致

`post2` 按 `GidResItem._ValNames` 从物理场 `_nodeRes` 取值。main 中注册的列名（如 `velU/accU`）必须与 `uPhy` 中 `_nodeRes["velU"]` 的键名**逐字符一致**，否则取到空向量越界。

### 5.11 pyTool 代码生成器会覆盖手写代码

`pyTool/test/testDEL2D.py` 是**代码生成器**（`MakerCpp.makeAll()`），运行会用模板重新生成 C++ 源码并覆盖手写文件，且其单元名为 `ElQ4g` 而非 `NewmarkQ4g`。**手写的 DEl2D 代码不可运行该脚本重新生成**，pyTool 仅用于生成 dat 数据文件。

### 5.12 编译编码：UTF-8 中文注释

源文件含 UTF-8 中文注释，MSVC 默认按系统代码页（GBK）解析会乱码并连锁报「标识符未声明」等虚假错误。顶层 CMake 已加：
```cmake
if(MSVC)
    add_compile_options(/utf-8)
endif()
```
新增带中文注释的源文件无需额外处理。

### 5.13 Newmark 参数 gamma/beta 改为前处理读取

`gamma/beta` 不再硬编码于 `NewmarkData::caculate`，改由 `NewmarkDispFieldData::_addParams = {{"newmark","gamma","beta"}}` 声明，`gidPrePost::pre()` 读 dat 的 `* name=newmark` 段（单星，与 time 段同构）回填到 `_paramValues`，`caculate` 用 `getParam("newmark",...)` 取值。dat 须含：

    * name=newmark,structure="F64"
    0.5 0.25

缺该段时 `getParam` 返回 0.0，`beta<=0` 会被 `caculate` 拦截报错。该机制由核心库 `_addParams`/`_paramValues`/`getParam` 通用支持，任意层（FEMData/物理场/单元）均可声明参数组。

端到端验证（`del2d.exe` 读 `del2d.dat`）：将 newmark 段改为 `0.5 0.0`（beta=0）运行，输出 `[NewmarkData] beta <= 0, 请检查 dat 文件 newmark 段` 并提前返回；改回 `0.5 0.25` 或 `0.6 0.4` 则通过校验进入求解，证明 beta 取自文件而非硬编码。机制本身的回归由 `testPreParam` 覆盖（验证三层回填与防御分支：组不存在/参数名不存在/向量越界均返回 0.0）。

> 已知限制（与本机制无关，待单独调试）：当前 `del2d.dat`（含完整 mat/elem/ubf 网格段）运行至 `PhyFieldData::initMatrix()` 会崩溃（`_kVar` 异常致 `std::length_error`），疑为 DEl2D 接入 GiD dat 读取链路的预存问题（readMate/coord 行推进或材料命名 `mat_<name>` vs `<name>`）。删除网格段仅保留 baseData/time/newmark/coord 时可正常走到参数校验与求解，故 newmark 参数读取机制本身不受影响——该机制已由上述 beta=0 拦截验证与 `testPreParam` 充分证明。

## 六、与 El2D（静力升级）的差异速查

| 方面 | El2D（静力） | DEl2D（动力 Newmark） |
|---|---|---|
| eProgram | 用基类 `eProgram_el` | **重写**（有效矩阵+载荷） |
| uPhy | 仅位移+应力 | 位移+速度+加速度+应力，并保存历史 |
| 单元 run | 仅 estif | estif + emass + edamp + eload |
| 物理场成员 | 无历史 | `_u/_v/_w` + `_u1/_v1/_w1` |
| caculate | 单次 init→eProgram→solve→uPhy | **时间步循环** |
| 求解次数 | 1 次 | nStep 次 |

---

## 七、结论

本次升级把 FEPG 生成的 C 动力求解器，改造为基于 CDFEG 库、符合 SOLID/KISS/DRY 的现代 C++ Newmark-β 动力示例。关键技术点在于：**物理场子类重写 eProgram 实现有效矩阵与有效载荷装配、重写 uPhy 实现状态更新与历史保存、数据类 caculate 实现时间步主循环**。最易出错的三个坑是：误用稳态 `eProgram_el`、边界条件缓存基线（`_bSavedData0`）、每步矩阵/载荷清零。
